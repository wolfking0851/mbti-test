## 2026-03-03 - Heartbeat 重复通知的预防措施

### 学到的经验

#### 1. 状态标记必须持久化
**问题：** 任务完成后，Heartbeat 仍每小时触发并发送通知

**解决方案：**
```python
# ✅ 检查是否已发送过通知
if progress.get("completionNotified", False):
    log(f"✅ 任务已完成且已通知，跳过 ({completed}/{total_tasks})")
    return

# ✅ 发送通知后立即标记
send_dingtalk_message(...)
progress["completionNotified"] = True
save_progress(progress)
```

**最佳实践：**
- ✅ 所有自动任务都应该有状态标记
- ✅ 状态必须持久化到文件
- ✅ 每次执行前先检查状态

#### 2. cron 配置必须唯一
**问题：** cron 配置中有重复任务，导致同一脚本同时运行两次

**解决方案：**
```bash
# ❌ 错误：重复配置
5 * * * * python3 script.py
5 * * * * python3 script.py

# ✅ 正确：单个配置
5 * * * * python3 script.py
```

**最佳实践：**
- ✅ 定期审计 crontab 配置（每月一次）
- ✅ 使用注释标记每个任务的用途
- ✅ 添加任务前检查是否已存在
- ✅ 使用 `crontab -l | grep <pattern>` 验证

#### 3. 修复问题必须彻底
**问题：** 第一次修复只处理了代码逻辑，未检查 cron 配置

**解决方案：**
**问题排查清单：**
```
□ 1. 检查代码逻辑（completionNotified）
□ 2. 检查 cron 配置（是否重复）
□ 3. 检查进程状态（是否有多个实例）
□ 4. 检查日志（确认修复效果）
□ 5. 验证下次触发（等待下一次 Heartbeat）
```

**最佳实践：**
- ✅ 修复后必须验证所有可能的原因
- ✅ 使用清单确保不遗漏
- ✅ 等待下一次触发验证修复效果

## 2026-03-03 - Gateway 性能优化

### 学到的经验

#### 1. 定期重启 Gateway
**问题：** Gateway 长时间运行导致：
- 内存累积（628MB）
- 上下文积累（100+ 条消息）
- 消息队列积压
- 响应速度变慢（3-8 分钟）

**解决方案：**
```bash
# 重启 Gateway
systemctl --user restart openclaw-gateway

# 验证
systemctl --user status openclaw-gateway
```

**最佳实践：**
- ✅ 每天重启一次 Gateway（建议凌晨 3 点）
- ✅ 内存超过 600MB 时重启
- ✅ 响应时间超过 30 秒时重启
- ✅ 收到重复消息时重启

#### 2. 监控响应时间
**问题：** 响应速度逐渐变慢，用户感知明显

**解决方案：**
```bash
# 查看最近任务的响应时间
tail -100 /tmp/openclaw/openclaw-2026-MM-DD.log | grep "durationMs"

# 正常响应时间
# - 简单问题：2-5 秒 ✅
# - 中等任务：10-30 秒 ✅
# - 复杂任务：30-60 秒 ⚠️
# - 超过 60 秒：❌ 需要优化
```

**最佳实践：**
- ✅ 每天检查响应时间日志
- ✅ 复杂任务分解为小任务
- ✅ 减少工具调用次数
- ✅ 批量操作代替单次调用

#### 3. 消息去重机制
**问题：** 同一条消息发送两次

**解决方案：**
- ✅ 重启 Gateway 清理消息队列
- ✅ 检查消息去重逻辑
- ✅ 添加消息 ID 追踪

**最佳实践：**
- ✅ 发送前检查消息 ID
- ✅ 使用消息队列确保幂等性
- ✅ 定期清理消息队列

## 2026-03-03 - 技能安装前的调研流程

### 学到的经验

#### 1. 技能调研清单
**安装前必须检查：**
```
□ 1. VirusTotal 评分（安全）
□ 2. GitHub 仓库（源码公开）
□ 3. 权限要求（避免系统权限）
□ 4. 依赖检查（是否需要额外安装）
□ 5. 功能对比（是否与现有技能重叠）
□ 6. 使用场景（实际使用频率）
□ 7. 风险评估（数据泄露、资源占用）
```

#### 2. 技能分类管理
**优先级分类：**
- ⭐⭐⭐⭐⭐ 高优先级（立即安装）
- ⭐⭐⭐⭐ 中优先级（调研后决定）
- ⭐⭐⭐ 低优先级（暂不需要）
- ❌ 不推荐（风险或不需要）

**分类标准：**
- 实用性（是否刚需）
- 风险等级（本地操作 vs 网络请求）
- 使用频率（每天 vs 偶尔）
- 功能重叠（是否与现有技能重复）

## 2026-02-28 - 邮件脚本管理优化

### 学到的经验

#### 1. 脚本路径管理
**问题：** 邮件脚本分散在多个目录，难以查找

**解决方案：**
- ✅ 建立统一的脚本目录结构
- ✅ 创建脚本索引文件
- ✅ 使用配置文件管理路径

**最佳实践：**
```
scripts/
├── email/
│   ├── send_email.py (通用邮件发送)
│   ├── send_task_summary.py (任务总结)
│   └── send_skill_check.py (技能检查)
├── backup/
│   ├── backup-manual.sh
│   └── backup-rotate.sh
└── weather/
    └── daily-weather-card.py
```

#### 2. 错误排查流程
**优化后的排查步骤：**
1. 先测试脚本是否可执行（直接运行）
2. 再检查脚本路径（使用 find 搜索）
3. 然后检查配置文件（SMTP 等）
4. 最后检查网络和权限

**命令模板：**
```bash
# 1. 直接测试
python3 reports/send_email.py

# 2. 搜索脚本
find . -name "*email*" | grep -v ".git"

# 3. 检查配置
cat config/email-config.json

# 4. 测试网络
telnet smtp.qq.com 465
```

#### 3. 配置与代码分离
**原则：** 配置信息应该独立于代码

**改进建议：**
- 将 SMTP 配置移到独立配置文件
- 使用环境变量或 JSON 配置文件
- 避免在代码中硬编码敏感信息

**示例配置：**
```json
{
  "smtp_server": "smtp.qq.com",
  "smtp_port": 465,
  "sender_email": "38132579@qq.com",
  "smtp_password": "${EMAIL_AUTH_CODE}"
}
```

#### 4. 错误日志记录
**必须记录的信息：**
- 错误发生时间
- 错误现象描述
- 排查过程和命令
- 根本原因分析
- 解决方案和验证

**记录位置：**
- `skills/self-improving-agent/.learnings/ERRORS.md`
- `memory/YYYY-MM-DD.md`
- `reports/error-log-*.md`
