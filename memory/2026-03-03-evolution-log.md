### 错误 1：六娃重复通知（严重）⭐⭐⭐⭐⭐

**问题：** 六娃每小时发送重复的完成通知

**影响：**
- ❌ 用户收到 4 次重复通知（19:05, 20:05, 21:05, 22:05）
- ❌ 每次发送 2 条相同消息（cron 重复导致）
- ❌ 总共 8 条重复消息

**根本原因：**
1. 代码没有 `completionNotified` 标记检查
2. cron 配置中有重复任务

**修复过程：**
1. ⏰ 13:38 - 第一次修复（添加 completionNotified 标记）
2. ⏰ 22:05 - 发现问题仍存在
3. ⏰ 22:10 - 第二次修复（删除重复 cron 任务）
4. ✅ 验证通过

**学到的经验：**
- ✅ 状态标记必须持久化
- ✅ cron 配置必须唯一
- ✅ 修复问题必须彻底（检查所有可能原因）

**预防措施：**
```python
# ✅ 标准模式
if progress.get("completionNotified", False):
    log(f"✅ 任务已完成且已通知，跳过")
    return

send_dingtalk_message(...)
progress["completionNotified"] = True
save_progress(progress)
```

```bash
# ✅ cron 配置审计（每月一次）
crontab -l | grep <pattern>
```

### 错误 2：响应速度变慢（严重）⭐⭐⭐⭐

**问题：** 响应时间从 2-5 秒变为 3-8 分钟

**影响：**
- ❌ 用户体验差（等待时间长）
- ❌ 资源浪费（双倍的 API 调用）
- ❌ 消息混乱（收到重复消息）

**根本原因：**
1. Gateway 长时间运行未重启（内存 628MB）
2. 上下文积累（100+ 条消息）
3. 工具调用过多（30-50 次）
4. 消息去重机制失效

**修复过程：**
1. 🔍 分析系统资源
2. 🔍 检查 OpenClaw 日志
3. 🔍 发现响应时间异常
4. ✅ 重启 Gateway
5. ✅ 内存恢复到 474MB
6. ✅ 响应速度恢复到 2-5 秒

**学到的经验：**
- ✅ 定期重启 Gateway（每天一次）
- ✅ 监控响应时间日志
- ✅ 内存超过 600MB 时重启
- ✅ 复杂任务分解为小任务

**预防措施：**
```bash
# ✅ 每天重启 Gateway（建议凌晨 3 点）
systemctl --user restart openclaw-gateway

# ✅ 监控响应时间
tail -100 /tmp/openclaw/openclaw-2026-MM-DD.log | grep "durationMs"
```
